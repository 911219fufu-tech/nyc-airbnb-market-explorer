# Final Report Plan

## Target Length

6 to 10 pages, excluding references if needed.

Recommended target:

- 8 pages main content

## Proposed Title

Interactive Web-based Visualization System for Exploring NYC Airbnb Market Dynamics

## Section Outline

### 1. Introduction and Motivation

Include:

- what problem the project solves
- why NYC Airbnb data matters
- who the target users are
- what decisions or questions the system supports

### 2. Related Work

Include:

- relevant visualization papers or systems
- any Airbnb, urban, geospatial, or time-series visualization references
- how the project differs from or extends prior work

Important:

- do not make this a paper summary list
- compare prior approaches to your own design choices

### 3. Data Description

Include:

- data source
- dataset size
- variables used
- temporal coverage
- spatial fields
- missing data and limitations
- preprocessing steps

### 4. Design and Methodology

Include:

- system goals
- visual encodings
- chart selection rationale
- layout strategy
- interaction design
- accessibility considerations
- alternative designs considered

Explicitly connect to course concepts:

- color
- perception
- encoding
- interaction

### 5. System Description / Implementation

Include:

- architecture overview
- frontend and backend stack
- data pipeline summary
- main components of the website
- important implementation details

### 6. Results and Insights

Include:

- concrete findings supported by the system
- screenshots only as evidence, not as the whole section
- interpretation of patterns

### 7. Limitations and Future Work

Include:

- data limitations
- design limitations
- interaction limitations
- possible extensions

### 8. Conclusion

Include:

- summary of contributions
- what the system enables users to understand

## Writing Standard

The report should:

- explain design decisions rather than just describing charts
- state at least three concrete insights
- use captions for figures
- keep terminology consistent
- avoid generic claims without evidence

## Figures to Prepare Later

- system overview screenshot
- one full dashboard screenshot
- one or two zoomed-in view examples
- architecture or data flow diagram

## Likely Weak Points to Avoid

- too much dataset description and too little design rationale
- screenshots without interpretation
- related work that is not connected to your final design
- vague statements such as "the chart is intuitive"
